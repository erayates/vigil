# Asset System

## Authoritative files

- `asset-registry.yaml` — status, provenance, permitted use and missing dependencies.
- `asset-manifest.json` — deterministic dimensions/checksums for current raster files.
- `../agent/asset-request-protocol.md` — mandatory stop-and-request workflow.
- `legio-focus-ui-reference.png` — user-supplied direction reference, not a shippable background.
- `ui-concept-v0.0.2.png` — current componentized implementation preview.
- `requests/` — request templates and pending user handoffs.

## Current state

The repository contains deterministic, original **scaffold-only** pixel assets generated by `scripts/generate_pixel_assets.py`. They allow agents to build window behavior, component layout, sprite state wiring and automated tests without downloading third-party art.

They are not final production art. An agent must not silently promote them to production status.

## Regeneration

```bash
npm run assets:generate
```

This regenerates scaffold assets and rebuilds `asset-manifest.json`.

Generated implementation paths:

```text
public/assets/brand/
public/assets/pixel/
public/assets/ui/frames/
public/assets/ui/textures/
src-tauri/icons/
```

Capture the browser-rendered documentation preview after a build:

```bash
npm run test:e2e:install
npm run preview:capture
```

A system Chromium can be selected with `CHROMIUM_PATH`.

## Visual reference handling

The supplied reference is analyzed into structure and tokens. It must not be used as one giant background with invisible HTML hotspots in production. Functional text, task content, timer values and controls remain live HTML.

## Request workflow

When a task needs a missing asset, create a request from `requests/ASSET-REQUEST-TEMPLATE.md`, cite the registry ID, ask the user and stop the task. Continue only after supply or an explicit placeholder waiver.
